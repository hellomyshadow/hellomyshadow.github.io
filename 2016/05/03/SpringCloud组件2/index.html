Hystrix

  1. Hystrix：Netflix公司的一款组件，是一种保护机制；
  2. Hystrix是Netflix开源的一个延迟和容错库，用于隔离访问远程服务、第三方库，防止出现级联失败；
  3. 雪崩问题：
    1. 微服务中，服务间的关系错综复杂，一个请求可能需要调用多个微服务接口才能实现，形成非常复杂的调用链路；
    2. 一旦被调用的服务提供者发生异常，请求阻塞，用户不会得到响应，那么Tomcat的这个线程也不会被释放，越来越多的用户请求到来时，随之越来越多的线程会阻塞，服务器支持的线程和并发数量是有限的，最后会导致服务器资源耗尽，从而会导致其他服务都不可用，这就是雪崩效应；
    3. 比如一个汽车生产线，如果某个零件因为种种原因无法供应，那么整台车就无法装配，陷入等待零件状态；如果很多车型都需要这个零件，那么整个车间都会陷入等待状态，导致所有生产都陷入瘫痪；
    4. Hystrix解决雪崩问题的手段主要有两种方式：线程隔离、服务熔断
  4. 线程隔离，服务降级
    1. 线程隔离：Hystrix为每个依赖服务调用分配一个小线程池，如果线程池已满，调用会被立即拒绝，默认不采用排队，加速失败的判定时间；
    2. 用户的请求不再直接访问服务，而是通过线程池中的空闲线程来访问，如果线程池已满或者请求超时，则会进行降级处理；
    3. 服务降级：优先保证核心服务，而非核心服务不可用或弱可用；
    4. 用户的请求故障时，不会被阻塞，更不会无限等待或者返回系统崩溃，至少可以看到一个执行结果，比如等待超过5s，则返回服务器繁忙的友好提示信息；
    5. 服务降级虽然导致请求失败，但不会导致阻塞，而且最多会影响这个依赖服务对应的线程池中的资源，对其他服务没有影响；
    6. 触发 Hystrix 服务降级的两种情况：线程池已满、请求超时

  5. 在服务消费方做降级处理：

    1. 引入依赖
      org.springframework.cloud
      spring-cloud-starter-netflix-hystrix

    2. 在启动类上注解 @EnableCircuitBreaker，启用Hystrix，注解 @EnableHystrix 也可以，但无法启用服务熔断
      @EnableCircuitBreaker
      @EnableDiscoveryClient
      @SpringBootApplication
      public class ConsumerApplication {

      @Bean
@LoadBalanced
public RestTemplate getRestTemplate() {
    return new RestTemplate();
}

public static void main(String[] args) {
    SpringApplication.run(ConsumerApplication.class, args);
}

      }

    3. 注解 @SpringCloudApplication 集成了 @EnableCircuitBreaker、@EnableDiscoveryClient、@SpringBootApplication
      @SpringCloudApplication
      public class ConsumerApplication {
      ...
      }

    4. Web层，在方法上注解 @HystrixCommand 开启降级处理，fallbackMethod属性指定错误处理的方法名
      @GetMapping(“{id}”)
      @HystrixCommand(fallbackMethod = “findByIdFallback”)
      public String findById(@PathVariable(“id”) Long id) {

      String url = "http://user-service/user/" + id;
String user = rest.getForObject(url, String.class);  //返回一个字符串，不再转为POJO对象
return user;

      }

      public String findByIdFallback(Long id) {

      return "服务器繁忙！"

      }

    5. 注意：熔断的的降级处理方法必须和正常请求的方法：参数列表和返回值保证一致；
    6. 在服务提供方加延迟，测试降级处理
      @Service
      public class UserService {
      @Autowired
private UserMapper mapper;
public User findById(Long id) {
    Thread.sleep(2000L);
    return mapper.selectByPrimaryKey(id);
}
      }
    7. 启用降级处理时，默认超过1s获取不到响应结果，就会触发服务降级；

    8. @DefaultProperties：注解在Controller类上，配置默认Fallback
      @RestController
      @RequestMapping(“consumer”)
      @DefaultProperties(defaultFallback = “defFallback”)
      public class ConsumerController {

      @Autowired
private RestTemplate rest;

@GetMapping("{id}")
@HystrixCommand  ----->启用降级
public String findById(@PathVariable("id") Long id) {
    String url = "http://user-service/user/" + id;
    String user = rest.getForObject(url, String.class);  //返回一个字符串，不再转为POJO对象
    return user;
}

public String defFallback() {
    return "服务器繁忙！"
}

      }

      1. 默认Fallback指向的方法不能有参数列表；
    9. 不同的业务，超时时间应该是不同的，@HystrixCommand的属性commandProperties
      @GetMapping(“{id}”)
      @HystrixCommand(commandProperties = {
      @HystrixPropery(name="execution.isolation.thread.timeoutInMilliseconds", value="2000") #2s超时
      })
      public String findById(@PathVariable(“id”) Long id) {
      ...
      }
      1. 全局配置默认超时时间：在消费者的配置文件中
        hystrix:
        command:
        default:
    execution:
        isolation:
            thread:
                timeoutInMilliseconds: 3000  #3s超时
      2. @HystrixCommand中不要再配置的超时属性；
      3. 还可以只针对某个调用的服务，甚至某个方法去配置超时时间
        hystrix:
        command:
        default:
    execution:
        isolation:
            thread:
                timeoutInMilliseconds: 3000  #3s超时
user-service:  #调用该服务时，启用降级处理的超时时间
    execution:
        isolation:
            thread:
                timeoutInMilliseconds: 3000
findById:  #调用该方法时，启用降级处理的超时时间
    execution:
        isolation:
            thread:
                timeoutInMilliseconds: 3000
  6. 服务熔断

1. 熔断器又叫断路器，Circult Breaker，类似家用的电路熔断器(保险丝)，如果电路发生短路，会立刻熔断电路，避免发生灾难；
2. 在分布式系统中，服务调用方可以自己进行判断某些服务反应慢或者存在大量超时的情况时，能够主动熔断，防止整个系统被拖垮；
3. 不同于保险丝不会重连，Hystrix可以实现弹性容错，当情况好转后，可以自动重连；
4. @EnableCircuitBreaker 同时会启用Hystrix熔断器；
5. Hystrix的熔断机模型
[模型]
    1. 三种状态：Closed、Open、Half Open
    2. Closed：熔断器关闭状态，所有请求都能正常访问；
    3. Open：熔断器打开，所有请求被降级处理；默认情况下，如果请求次数超过20，且失败比例达到50%(阈值)，则打开熔断器；
    4. Half Open：熔断器处于半开状态，Closed状态不是永久的，关闭后会进入休眠状态(默认5s)，称为休眠时间窗，随后熔断器会自动进入半开状态，放进一部分请求通过，测试该服务是否正常，如果请求仍然失败，则重新进入Open状态，直到放进的请求完全成功，才会进入Closed状态
6. 测试
    1. 服务提供者：取消的2s延迟
        @Service
        public class UserService {
            @Autowired
            private UserMapper mapper;
            public User findById(Long id) {
                return mapper.selectByPrimaryKey(id);
            }
        }
    2. 在服务消费者中，通过抛出异常的方式，手动控制熔断器断开，并且配置熔断器的参数
    @GetMapping("{id}")
    @HystrixCommand(commandProperties = {
        @HystrixProperty(name="circuitBreaker.requestVolumeThreshold", value="10"),  #10次请求，默认20
        @HystrixProperty(name="circuitBreaker.errorThresholdPercentage", value="60"), #失败比率60%，默认50
        @HystrixProperty(name="circuitBreaker.sleepWindowInMilliseconds", value="10000"),  #休眠时间10s，默认5s
    })  --> 对同一个服务发起10次请求，超过6次都是失败的，则开启熔断器，休眠10s后，进入半开状态，放进一部分请求进行测试
    public String findById(@PathVariable("id") Long id) {
        if(id%2 == 0) {
            throw new RuntimeException(""); //抛出异常会直接触发熔断
        }
        String url = "http://user-service/user/" + id;
        String user = rest.getForObject(url, String.class);
        return user;
    }
    3. 测试：
        http://localhost:8080/consumer/1  -->正常返回
        http://localhost:8080/consumer/2 --> 抛出异常，服务降级处理，返回"服务器繁忙！"
        快速访问 http://localhost:8080/consumer/2 超过10次，然后再立即访问 http://localhost:8080/consumer/1
        服务也会被降级处理，返回 "服务器繁忙！"，10s后又可以正常获取数据

Feign

  1. Feign：简化服务调用，可以把Rest的请求进行隐藏，伪装成类似SpringMVC的Controller，无需再拼接URL、参数等等；
    String url = "http://user-service/user/" + id;
String user = rest.getForObject(url, String.class);

  2. 服务消费者中：

    1. 引入依赖
      org.springframework.cloud
      spring-cloud-starter-openfeign
    2. Feign内部已经依赖了Ribbon和Hystrix，所以在 pom.xml 不用手动引入Ribbon依赖，但Hystrix依赖最好还是要引入；
    3. 在启动器类上注解 @EnableFeignClients，启用Feign，并去除RestTemplate的注册
      @EnableFeignClients
      @SpringCloudApplication
      public class ConsumerApplication {
      public static void main(String[] args) {
    SpringApplication.run(ConsumerApplication.class, args);
}
      }
    4. Feign客户端接口
      @FeignClient(“user-service”)
      public interface UserClient {
      @GetMapping("user/{id}")  -->请求资源路径
User findById(@PathVariable("id") Long id);
      }
      1. @FeignClient：声明Feign客户端，指定服务ID，再交给Ribbon实现负载均衡；
      2. 内部完全使用 SpringMVC的注解，Feign会根据注解生成URL；
      3. Feign会通过动态代理，为Feign客户端接口生成实现类，类似Mybatis的mapper

    5. 使用Feign的客户端发起请求
      @RestController
      @RequestMapping(“consumer”)
      public class ConsumerController {

      @Autowired
private UserClient client;

@GetMapping("{id}")
public User findById(@PathVariable("id") Long id) {
    User user = client.findById(id);
    return user;
}

      }

  3. Feign内部集成了Ribbon依赖和自动配置，配置Ribbon的超时时间
    ribbon:
    ConnectTimeout: 500  #连接超时时间为500ms，超过500ms没有连接上服务，则抛出异常
ReadTimeout: 2000  #读取数据的超时时间2s，超过2s还没有获取到响应，则抛出异常
  4. Feign集成的Hystrix使用方式
    1. 默认是关闭的，在配置文件中开启
      feign:
      hystrix:
      enabled: true  #开启Feign的熔断功能
    2. 在注解 @FeignClient 中配置属性 falback，属性值是一个类.class，且这个类要实现注解的接口，用于处理服务降级
      @FeignClient(value = “user-service”, fallback = UserClientFallback.class)
      public interface UserClient {
      @GetMapping("user/{id}")  -->请求资源路径
User findById(@PathVariable("id") Long id);
      }
      @Component –>注入到Spring中
      public class UserClientFallback implements UserClient {
      public User findById(Long id) {  //服务降级的处理
    User u = new User();
    u.setId(id);
    u.setName("未知用户！");
    return u;  //响应数据
}
      }
    3. 关闭 user-service 服务，访问服务消费者：http://localhost:8080/consumer/1
  5. 请求压缩
    1. Feign支持对请求和响应进行GZIP压缩，以减少通信过程中的性能损耗；
    2. 开启压缩
      feign:
      compression:
      request:
    enabled: true  #开启请求压缩
    mime-types: text/html,application/xml,application/json  #设置压缩的数据类型
    min-request-size: 2048  # 触发压缩的大小下限
response:
    enabled: true  #开启响应压缩
  6. 日志级别
    1. 通过 logging.level.xx=debug 来设置日志级别，但是它对Feign客户端没有效果，因为 @FeignClient 注解的客户端在动态代理时，都会生成一个新的 Feign.Logger 实例，需要额外指定这个日志级别
    2. 设置 cn.cloud 包下的日志级别都是 debug
      logging:
      level:
    cn.cloud: debug
    3. 使用配置类定义日志级别
      @Configuration
      public class FeignConf {
      @Bean
Logger.Level feignLoggerLevel() {
    return Logger.Level.FULL;  //设置日志级别为FULL，另外还有NONE、BASIC、HEADERS...
}
      }
    4. NONE：不记录任何日志信息，默认值；
    5. BASIC：仅记录请求的方法、URL以及响应状态码和执行时间；
    6. HEADERS：在BASIC的基础上，额外记录了请求和响应的头信息；
    7. 在Feign客户端的注解上，指定配置类
      @FeignClient(value = “user-service”, fallback = UserClientFallback.class, configuration = FeignConf.class)
      public interface UserClient {
      @GetMapping(“user/{id}”) –>请求资源路径
      User findById(@PathVariable(“id”) Long id);
      }
