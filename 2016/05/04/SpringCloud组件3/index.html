网关之Zuul

  1. Zuul：Netflix开源的微服务网关，它可以和Eureka、Ribbon、Hystrix等组件配合使用；
  2. Zuul的核心是一系列的过滤器，这些过滤器的功能：

1. 身份认证与安全：识别每个资源的验证要求，并拒绝那些与要求不符的请求；
2. 审查与监控：在边缘位置追踪有意义的数据和统计结果，从而带来精确的生产视图；
3. 动态路由：动态地将请求路由到不同的后端集群；
4. 压力测试：逐渐增加指向集群的流量，以了解性能；
5. 负载分配：为每一种负载类型分配对应容量，并弃用超过限定值的请求；
6. 静态响应处理：在边缘位置直接建立部分响应，从而避免转发到内部集群；
7. 多区域弹性：跨越AWS Region 进行数据路由，旨在实现ELB使用的多样化，以及让系统的边缘更贴近系统的使用者；
  1. SpringCloud对Zuul做了整合与增强；目前Zuul使用的默认HTTP客户端是Apache HttpClient，
    也可以使用RestClient，okhttp3.OkHttpClient
    1. 启用RestClient时，配置 ribbon.restclient.enabled=true
    2. 启用okhttp3.OkHttpClient，则配置 ribbon.okhttp.enabled=true
  2. 网关就是整个系统的入口，如果没有网关进行控制，外界可以访问系统中的任何服务，甚至可以把自己的服务注册到系统的注册中心；
  3. 网关的核心功能：过滤和路由
  4. Zuul加入后的架构
    [图片]
    1. 不管是来自客户端的请求，还是服务内部的调用，一切对服务的请求都会经过Zuul，然后再由网关来实现鉴权、动态路由等操作；
    2. Zuul就是服务的统一入口；

  5. 在父工程下创建一个子工程gateway

    1. 引入依赖
      org.springframework.cloud
      spring-cloud-starter-netflix-zuul
    2. Zuul依赖中已经引入了 web、hystrix、ribbon 等依赖
    3. 创建启动类，注解 @EnableZuulProxy，而不是 @EnableZuulServer，因为它的功能不完善，太单一；
      @EnableZuulProxy
      @SpringBootApplication
      public class GatewayApplication {
      ...
      }

    4. 配置
      server:

      port: 8090

      spring:

      application:
    name: gateway

      zuul:

      routes:  #配置路由规则
    hehe: #路由ID，值是key-value，作为一条路由规则
        path: /user-service/**   #匹配的路径，以/user-service开头的所有路径
        url: http://127.0.0.1:8081  #转发的路径，即转发的服务提供者

    5. /user-service 只作为匹配路径的前缀，** 才会被转发
      http://127.0.0.1:8090/user-service/user/9 –> http://127.0.0.1:8081/user/9

  6. 使用Eureka的客户端依赖，注册服务，拉取服务列表

    1. 引入依赖
      org.springframework.cloud
      spring-cloud-starter-netflix-eureka-client
    2. 配置
      eureka:
      client:
    service-url:
        defaultZone: http://127.0.0.1:8761/eureka
      zuul:
      routes:
    hehe:
        path: /user-service/**
        serviceId: user-service  #指定要转发的服务ID
    3. Zull底层根据配置的 serviceId，使用 EurekaClient 获取服务列表，通过Ribbon的负载均衡算法，获取一个服务实例；
    4. 测试：http://127.0.0.1:8090/user-service/user/9
    5. 既然路由ID可以自由命名，那么就简化配置方案：
      zuul:
      routes:
    user-service: /user-service/**  #key是转发的服务ID，value是匹配的路径
    6. 事实上，Zuul默认就是这么配置的，它会拉取所有的服务列表，把它们都默认配置为：服务ID: /服务ID/**
    7. 也就是说，如果不需要更改映射路径，zuul.routes 可以不用配置，但即使手动更改了映射路径，/服务ID/** 的方式依然有效
      zuul:
      routes:
    user-service: /user/**  # 映射匹配的路径为 /user
      http://127.0.0.1:8090/user/user/9 等效于 http://127.0.0.1:8090/user-service/user/9
    8. 当然，有些服务不想让 Zuul 自动配置，则让Zuul忽略：
      zuul:
      ignored-services:  #忽略注册的服务
    - consumer-service
      http://127.0.0.1:8090/consumer-service/consumer/9 不可以再访问

    9. 转发前缀：匹配路径的前缀默认是不转发给微服务的，strip-prefix 属性可以控制是否转发，默认值是true
      zuul:

      routes:
    user-service:
        path: /user/**
        serviceId: user-service
        strip-prefix: false  #转发前缀
ignored-services:
    - consumer-service
      1. 访问：http://127.0.0.1:8090/user/9 –> http://127.0.0.1:8081/user/9
        测试发现：带有服务ID的前缀 http://127.0.0.1:8090/user-service/user/9 依然可以访问
      2. 转发前缀的全局配置：zuul.strip-prefix
        zuul:
        routes:
    user-service:
        path: /user/**
        serviceId: user-service
strip-prefix: false
然而，测试发现：并不能去掉前缀，访问时仍然要加上前缀 http://127.0.0.1:8090/user/user/9

      3. 如果路由规则是简化配置，zuul.strip-prefix属性无效。
        zuul:

        routes:
    user-service: /user/**
strip-prefix: false  # 无效，不会转发前缀

        http://127.0.0.1:8090/user/user/9 –> http://127.0.0.1:8081/user/9

    10. 全局添加前缀：所有请求的前面都必须加上的前缀
      zuul:

      prefix: /api  #全局前缀
routes:
    user-service: /user/**  # /user 是局部前缀

      http://127.0.0.1:8090/api/user/user/9 –> http://127.0.0.1:8081/user/9

      1. 局部 strip-prefix=false 只对局部前缀有效，不会转发全局前缀
        zuul:
        prefix: /api
        routes:
        user-service:
    path: /user/**
    serviceId: user-service
    strip-prefix: false
        http://127.0.0.1:8090/api/user/9 –> http://127.0.0.1:8081/user/9

        过滤器

  1. Zuul作为网关的一个重要功能，就是实现请求的鉴权(权限控制)，通过Zuul提供的过滤器实现；
  2. ZuulFilter 是过滤器的顶级父类，最重要的方法有4个
    public abstract ZuulFilter implements IZuulFilter {
    abstract public String filterType();
abstract public int filterOrder(); 
boolean shouldFilter();  //来自IZuulFilter
Object run() throws ZuulException;  //来自IZuulFilter
......
    }
    1. filterType()：过滤器的类型，4种类型
      1. pre：请求在被路由之前执行，前置过滤器
      2. routing：在路由请求时执行
      3. post：在routing和error过滤器之后调用，后置过滤器
      4. error：处理请求时发生错误调用，异常处理器
    2. filterOrder()：过滤器的执行顺序，数值越小，优先级越高
    3. shouldFilter()：是否过滤
    4. run()：过滤器执行的逻辑
  3. 过滤器的执行生命周期
    [图片]
    1. 正常流程：请求到达 –> pre Filter –> routing Filter –> 进行路由 –> 请求到达真正的服务提供者 –>
      返回处理结果 –> post Filter –> 响应数据
    2. 异常流程：
      1. 在整个过程中，pre/routing Filter出现异常 –> error Filter 处理完毕后，将请求交给 post Filter –> 响应数据
      2. 如果 error Filter 出现异常，最终也会进入post Filter，然后响应数据；
      3. 如果 post Filter 出现异常，会进入 error Filter，但与 pre/routing Filter 不同，请求不会再到达 post Filter了
    3. 前置过滤器的应用：权限控制，限流(限制流量，限制某个用户在一段时间内只能访问一次，限制最多只能有1000个人访问等等)
    4. 后置处理器和异常处理器通常结合使用，做一些通用的异常处理；
    5. 服务调用时长统计：pre 和 post Filter 结合使用。
  4. Zuul内置了很多过滤器，也允许自定义过滤器，比如模拟登录的校验，如果请求中有 access-token 参数，则放行
    1. 在网关模块gateway中，自定义过滤器
      @Component –>加入Spring IoC容器
      public class LoginFilter extends ZuulFilter {
      public String filterType() {
      return FilterConstants.PRE_TYPE;  // 前置过滤器
      }
      public int filterOrder() {
      return FilterConstants.PRE_DECORATION_FILTER_ORDER -  1;  //等待把请求参数处理完，再做拦截
      }
      public boolean shouldFilter() {
      return true;
      }
      public Object run() throws ZuulException {
      //1. 在内置的过滤器中，查看如何获取Request对象
//1-1. 获取请求上下文，生命周期：一次请求，即从请求到达Zuul，到最终响应数据；只限Zuul中使用，不会带到微服务中
RequestContext ctx = RequestContext.getCurrentContext();
//1-2. 获取当前的请求对象
HttpServletRequest request = ctx.getRequest();
//2. 获取请求参数
String token = request.getParameter("access-token");
//3. 判断是否存在 access-token 
if(StringUtils.isBlank(token)) {  --->引入Apache的工具类依赖：commons-lang3
    //trim() 内部会多次截取字符串，效率非常差，所以StringUtils.isBlank()内部没用trim()
    //不存在，则认为没有登录，拦截请求
    ctx.setSendZuulResponse(false); //拦截
    ctx.setResponseStatusCode(HttpStatus.FORBIDDEN.value());  //设置响应状态码403
}
return null;
      }
      }
    2. 测试
      http://127.0.0.1:8090/user/9 –> 请求被拒绝，403
      http://127.0.0.1:8090/user/9?access-token=123 –> http://127.0.0.1:8081/user/9
  5. 负载均衡和服务熔断
    1. Zuul默认集成了Ribbon负载均衡和Hystrix熔断机制，但所有的超时策略都是默认值，比如熔断超时时间为1s，很容易触发
    2. 手动配置
      hystrix:
      command:
    default:
        execution:
            isolation:
                thread:
                    timeoutInMilliseconds: 6000  #6s超时
      ribbon:
      ConnectTimeout: 500  #默认1000
ReadTimeout: 2000  #默认1000
MaxAutoRetriesNextServer: 1  # 不同服务实例的重试次数，一个服务多个实例，在不同服务实例之间重试，默认为1
MaxAutoRetries: 0  # 对当前服务实例的重试次数，默认为0，不重试
    3. 内部在计算时，Ribbon的超时总时长 不能大于 hystrix熔断的超时时长，否则Ribbon还没有超时，服务就已经熔断了！
    4. Ribbon超时总时长的计算规则：(ConnectTimeout+ReadTimeout)(MaxAutoRetries+1)(MaxAutoRetriesNextServer+1)
    5. 所以默认情况下，Ribbon超时总时长=(1000+1000)(0+1)(1+1)=4000ms
  6. Zuul的高可用性
    1. Zuul也是一个服务，那么就可以启用多个Zuul服务，自动注册到Eureka，形成集群；
    2. 如果是服务内部访问，访问Zuul，自动负载均衡，没有问题；但是，Zuul更多是外部访问，如PC、移动端，它们并不知道有这么多网关地址，无法通过Eureka进行负载均衡；
    3. 此时就会使用其他的服务网关，来对Zuul进行代理，比如Nginx
      用户请求 –> Nginx –> Zuul-1、Zuul-2 … –> Service-1、Service-2 …
  7. Eureka、Ribbon、Hystrix、Feign、Zuul 这些只是SpringCloud的一小部分，还有更多组件
  8. spring-cloud-config：统一配置中心，让微服务自动从Git上拉取最新的配置，并缓存，不再手动更改配置文件；
    使用Git的Webhook钩子，还可以实现主动通知配置中心，告知配置发生了变化，那么配置中心会通过消息总线、通知所有的微服务、去更新配置；
  9. spring-cloud-bus：消息总线，底层是消息队列，RabbitMQ或kafuka
  10. spring-cloud-stream：消息通信
  11. spring-cloud-hystrix-dashboard：容错统计，形成图形化界面
  12. spring-cloud-sleuth：链路追踪，结合Zipkin，便于追踪服务的错误
